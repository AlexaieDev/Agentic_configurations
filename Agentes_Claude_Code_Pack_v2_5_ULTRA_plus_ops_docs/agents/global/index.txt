INDEX.txt — Catálogo de Agentes por Tamaño del Equipo y Madurez

PROPÓSITO
Este índice te ayuda a seleccionar qué agentes usar según:
- Producto y plataforma (Web, Mobile, Desktop, Cloud)
- Tamaño del equipo
- Nivel de madurez (startup vs scale-up vs enterprise)
- Objetivo principal (velocidad, calidad, confiabilidad, gobernanza)

Cómo usar este índice
1) Parte con el “núcleo mínimo” de tu plataforma principal.
2) Agrega agentes transversales de Code Health tan pronto tengas 2+ repos activos.
3) Incorpora agentes de Cloud/SRE/Security cuando tu sistema tenga:
   - tráfico real,
   - integración con pagos/datos sensibles,
   - varias squad/streams,
   - o costo cloud relevante.
4) Usa Technology Critic cuando la organización:
   - esté creciendo,
   - haya señales de “framework sprawl”,
   - o planee una migración grande.

============================================================
C) CUÁNDO ACTIVAR CADA AGENTE (SEÑALES)
============================================================

Bug Hunter Agent
Úsalo cuando:
- hay bugs recurrentes en prod,
- el equipo está creciendo,
- necesitas fijar estándar de test de regresión.

Refactor & Code Quality Agent
Úsalo cuando:
- ves duplicación frecuente,
- PRs se vuelven difíciles de revisar,
- cambias features y rompes otras sin querer.

Performance & Efficiency Agent
Úsalo cuando:
- Core Web Vitals bajan,
- hay quejas de latencia,
- el costo cloud sube sin explicación.

Test Strategy Agent
Úsalo cuando:
- hay debates sobre “qué testear”,
- E2E se vuelve lento e inestable,
- necesitas balance de costo/impacto.

Technology Critic & Improvement Agent
Úsalo cuando:
- aparece “framework sprawl”,
- quieres introducir una nueva plataforma,
- planeas migraciones grandes.

Observability Agent
Úsalo cuando:
- ya hay usuarios reales,
- necesitas reducir MTTR,
- quieres SLOs medibles.

Cloud Security Agent
Úsalo cuando:
- manejas datos sensibles,
- trabajas en entornos regulados,
- hay múltiples equipos desplegando a cloud.

DX Agent
Úsalo cuando:
- el onboarding tarda demasiado,
- cada repo tiene configuración distinta,
- builds y tests son lentos por desorden de tooling.

D) PROPUESTA DE ADOPCIÓN EN 4 FASES
============================================================

Fase 1 — “Ship fast con higiene”
- Agentes core de tu plataforma
- Web/Mobile CI/CD
- Bug Hunter
- Test Strategy

Fase 2 — “No dejes crecer la deuda”
- Refactor & Code Quality
- DX
- QA por plataforma

Fase 3 — “Operar con datos”
- Observability
- Performance & Efficiency
- Cloud Architecture + Platform/DevOps

Fase 4 — “Escala organizacional”
- Technology Critic
- SRE
- Cloud Security
- Políticas globales obligatorias


============================================================
E) NOTAS DE DISEÑO DE AGENTES (BUENAS PRÁCTICAS)
============================================================

- Mantén agentes con una responsabilidad principal.
- Evita solapamientos: cuando haya dudas, Architecture define estándar y los demás lo implementan.
- Haz que todos respeten la política global.
- Prefiere outputs cortos y accionables:
  - checklist,
  - ADR breve,
  - PR sugerida,
  - test de regresión.
- Todo agente debe favorecer:
  - reutilización modular,
  - contratos explícitos,
  - seguridad por defecto,
  - observabilidad mínima,
  - simplicidad como criterio fundamental.
